Step 1 â€” Installing and Setting up Apache

From my limited understanding, since I don't use LAMP that much, Apache is a web server that accepts and handles incoming requests, similar to express (nodejs) or fastapi (python)

sudo apt update
sudo apt install apache2
sudo ufw app list
sudo ufw allow ssh
sudo ufw allow in "Apache"

// verify changes
sudo ufw status

From my understanding, executing the commands above allows Apache to run within the VM's firewall, thus allowing Apache to function as a web server to serve incoming HTTP requests using port 80.

Step 2 â€” Installing MySQL

Next, we were tasked with setting up a relational database called MySQL. I have personally never used MySQL before since I was familiar with PostgreSQL using ORMs. However, since both are SQL-based databases, the experience should be familiar

sudo apt install mysql-server
sudo apt install mysql-server
sudo mysql

For this project, I decided not to enable the VALIDATE PASSWORD plugin since security was not the priority of the stack

However, this initial setup requires that all queries be executed under root priveleges, which is not ideal for most applications under the principle of least-trust. Thus, following the guide, I created a subuser called example_user to simulate a user (or application) that is accessing the database

mysql> CREATE DATABASE example_database;
mysql> CREATE USER 'example_user'@'%' IDENTIFIED WITH mysql_native_password BY 'password';
mysql> GRANT ALL ON example_database.* TO 'example_user'@'%';

// login with sub-user account
mysql -u example_user -p

Using this user, I was able to experiment with the different basic MySQL queries (CREATE, INSERT, SELECT, DELETE, UPDATE) to interface with the database, as seen below:

mysql> CREATE TABLE example_database.todo_list (
item_id INT AUTO_INCREMENT,
content VARCHAR(255),
PRIMARY KEY(item_id)
);
mysql> INSERT INTO example_database.todo_list (content) VALUES ("My first important item");
mysql> SELECT * FROM example_database.todo_list;

With this, I was able to setup a preliminary structure for my database, which will be used later for my Todolist app

Step 4 â€” Creating a Virtual Host for your Website

This part seemed difficult at first since I was not familiar with the concept of Virtual Hosts but after using the guide, I was able to understand the surface-level of how it worked.


The main idea is creating a root directory for the website, and then setting up the configuration in order to host a website application from that root directory (and the assets and file structure herein).

sudo mkdir /var/www/your_domain
sudo a2dissite 000-default
sudo nano /etc/apache2/sites-available/your_domain.conf

There was an unforeseen bug here because of the difference in configuration in the EEEI VMs from a standard installation of Ubuntu as shown in the DO guide. In order to set the correct group ID, I used the groups command to view the user groups in the VM and use that for setting the group

sudo a2ensite your_domain
sudo chown -R $USER:$GROUP /var/www/your_domain
sudo apache2ctl configtest
sudo systemctl reload apache2
nano /var/www/your_domain/index.html

From here, the project directory was set at /var/www/reddie.dev and I was able to start working on the other aspects of the Lab Exercise by continuously adding more assets (.html and .php files)

Step 7 â€” Creating a Self-signed SSL Certificate to support HTTPS

Finally, I used HTML with TailwindCSS in order to create this documentation for my LAMP application

sudo ufw allow "Apache Full"
sudo a2enmod ssl
// restart apache to apply changes
sudo systemctl restart apache2
sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 -keyout /etc/ssl/private/apache-selfsigned.key -out /etc/ssl/certs/apache-selfsigned.crt

Following the steps in the guide, I was able to generate an SSL certificate

I then applied the configuration settings below in order to make Apache support HTTPS as well as redirect all HTTP traffic to use HTTPS instead

sudo nano /etc/apache2/sites-available/10.158.90.45.conf
// verify and restart apache to apply changes
sudo apachectl configtest sudo systemctl reload apache2

Unfortunately, since the SSL cert was only self-signed, it won't show up as "trusted" under modern browsers. This is because browsers need SSL certs to be verified by a Certificate Authority such as Let's Encrypt

Step 8 â€” Installing Wordpress in a /blog subdirectory

In order to make adding more content easier in the future (since creating HTML from scratch is not a sustainable and scalable solution - especially for multiple users), I added a Content Management System (CMS), Wordpress, to my site using this guide

sudo mysql -u root
mysql> ALTER USER 'root'@'localhost' IDENTIFIED WITH mysql_native_password BY 'password';
mysql -u root -p
mysql> CREATE DATABASE wordpress DEFAULT CHARACTER SET utf8 COLLATE utf8_unicode_ci;
mysql> CREATE USER 'wordpressuser'@'%' IDENTIFIED WITH mysql_native_password BY 'password';
mysql> GRANT ALL ON wordpress.* TO 'wordpressuser'@'%';
mysql> FLUSH PRIVILEGES;
mysql> EXIT;

First, I prepared the MySQL database server by adding a specific database for wordpress

sudo apt update
sudo apt install php-curl php-gd php-mbstring php-xml php-xmlrpc php-soap php-intl php-zip
sudo systemctl restart apache2

After that, I installed some php extensions which were needed by wordpress to function. Afterwards, I edited the Apache2 Virtual Host configuration to allow support for Wordpress. I spent a lot of time debugging this setup since I can't figure out why the site was not loading. Turns out, I just need to modify a portion of the virtual host config - not replacing the entire config file.

sudo nano /etc/apache2/sites-available/your_domain.conf

After applying changes, as well as allowing the rewrite plugin, the file directory is ready for installing wordpress

sudo a2enmod rewrite
sudo sudo apache2ctl configtest
sudo sudo a2ensite your_domain.conf
sudo sudo systemctl restart apache2

Installing wordpress was a breeze since all of the core files are neatly available in a zipped file. Simply extracting this to the root directory installs Wordpress

cd /tmp
curl -O https://wordpress.org/latest.tar.gz
tar xzvf latest.tar.gz
touch /tmp/wordpress/.htaccess
cp /tmp/wordpress/wp-config-sample.php /tmp/wordpress/wp-config.php
mkdir /tmp/wordpress/wp-content/upgrade
sudo cp -a /tmp/wordpress/. /var/www/your_domain

However, since Wordpress is a CMS, I needed to provide ownership access to the Apache2 and Wordpress Ubuntu User in order to allow WP to modify the system files. This proved problematic on my end since I had to change the root directory owner between myself and www-data in order to modify the files

sudo chown -R www-data:www-data /var/www/your_domain
sudo find /var/www/your_domain/ -type d -exec chmod 750 {} \;
sudo find /var/www/your_domain/ -type f -exec chmod 640 {} \;

Afterwards, I just needed to set the correct configuration settings in the wp-config.php file in order to finally get in the initial startup Wordpress page

curl -s https://api.wordpress.org/secret-key/1.1/salt/
sudo nano /var/www/your_domain/wp-config.php
